测试覆盖率增强技巧

核心思路是：从 “功能点”、”分支条件”、”边界场景” 三个维度度全面覆盖，利用工具定位盲区，结合项目实际逻辑补充测试

测试所有条件分支;测试错误处理;函数覆盖率（工具函数、事件处理函数、钩子函数）;”正常场景” 外的边界情况（如极限值、空值、错误输入）

注意事项：避免 “为覆盖率而测试”
提高覆盖率的同时，需避免盲目追求数字而编写写无意义的测试：
不测试明显不会逻辑的代码（如简单的 getter/setter）
不重复测试相同逻辑（如多个测试用例覆盖同一分支）
优先覆盖核心逻辑（如绘图核心逻辑），再非边缘的辅助代码

间谍（Spies）与模拟（Mocks）

理解：

  1. 间谍（Spies）是 Jest 提供的一种特殊函数，用于监视其他函数的调用情况：
    记录函数被调用的次数
    记录函数被调用时的参数
    默认情况下不影响原函数的正常执行
  2. 模拟（Mocks）是 Jest 提供的一种 “假函数”，用于替代真实函数：
    完全替换原函数，返回预设值
    记录自身被调用的情况
    用于隔离测试，避免依赖外部资源

间谍与模拟的区别

间谍：只监视，不改变原函数行为（监控 Canvas 上下文方法调用）
模拟：完全替换原函数，可自定义返回值（较少使用，除非需要替换复杂逻辑）

为什么在 Canvas 测试中需要它们

在Canvas 绘图应用中，无法直接 “看到” 绘制的图形，但可以通过监视Canvas上下文的方法调用情况，来验证绘图功能是否正常工作

例如，当调用 context.fillRect() 时，间谍可以告诉我们：
这个方法是否被调用了；调用时使用了哪些参数（位置、大小等）；调用了多少次

语法

  1. 创建间谍
    const fillRectSpy = jest.spyOn(ctx, 'fillRect');

    ctx 是你获取的 Canvas 2D 上下文
    ‘fillRect’ 是你要监视的方法名
    这个操作不会改变 fillRect 的原有功能，只是增加了监控

  2. 断言间谍记录
  3. 清理间谍
    fillRectSpy.mockRestore();

canvas测试–豆包老师版：

思路

关注组件和canvas的交互逻辑，不直接测绘图细节（比如线条颜色、坐标），去测试用户操作->鼠标事件->的否正确调用了Canvas api中方法；Canvas 操作后的副作用（如状态更新、事件触发）是否符合预期

组件初始化验证：确认两个 Canvas 元素正确渲染，上下文正常获取
背景 Canvas 测试：验证背景绘制功能正常工作
工具选择测试：检查不同绘图工具（矩形、圆形等）的切换逻辑
交互功能测试：鼠标绘制基本图形,标签和文字输入功能,颜色和字体大小选择
事件响应测试：确保鼠标事件（点击、拖拽、右键）正确触发相应功能

canvas测试–DeepSeek老师版：

思路

由于不能模拟canvas，我们将使用jest的spyOn来监听canvas的上下文方法，并验证这些方法是否被正确调用。
请注意，我们不会真正模拟CanvasRenderingContext2D的所有方法，而是使用jest.fn()来创建模拟函数，以便跟踪调用情况。
由于有两个canvas，我们需要分别获取它们的上下文，并分别进行模拟。

注意：由于我们无法真正模拟用户事件，我们将通过模拟事件处理函数来测试事件处理逻辑。
在测试中，我们不会真正渲染React组件，而是使用@testing-library/react来渲染组件，并获取canvas元素。
但是，由于我们要求不要模拟canvas，实际上我们不会使用jest-canvas-mock（虽然它通常用于解决canvas在jest中的兼容问题），而是采用监听上下文方法的方式。
我们将使用@testing-library/react来渲染组件，并模拟用户事件。
步骤：

  1. 安装必要的测试库：@testing-library/react, @testing-library/user-event（用于模拟事件，但也可以使用@testing-library/react的fireEvent）
  2. 编写测试用例。
    由于没有具体组件代码，我将提供一个测试文件的大致结构，并包含详细的注释。
  • 思路：
  1. 设置测试环境：创建两个canvas元素，并添加到document.body中（在测试结束后需要清理）。
  2. 获取两个canvas的2D上下文，并使用jest.spyOn模拟所有需要的方法（如fillRect, strokeRect, beginPath, arc, fillText等）。
  3. 模拟事件：如mousedown, mousemove, mouseup, contextmenu等，并传递模拟的事件对象。
  4. 测试各个功能：
    • 绘制矩形：模拟鼠标事件，检查strokeRect或fillRect是否被调用（取决于当前是描边还是填充）。
    • 绘制圆形：同样模拟鼠标事件，检查arc和stroke或fill方法是否被调用。
    • 填充矩形：同上，但检查fillRect。
    • 文本输入：模拟右键菜单事件，然后模拟输入框输入，再检查fillText是否被正确调用。
    • 颜色和字体大小：改变颜色和字体大小的选择，然后进行绘制操作，检查绘制时是否使用了正确的颜色和字体。
      由于代码结构未知，以下是一个假设的组件结构，我们将编写一个测试文件。
      假设组件：
    • 有两个canvas，id分别为”background-canvas”和”drawing-canvas”
    • 组件中实现了鼠标事件处理函数，并绑定在drawing-canvas上
